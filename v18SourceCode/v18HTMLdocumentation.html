v18 page shell HTML documentation

Converting a v17 page to a v18 page requires some minimal changes to both your HTML and JS code. For convenience I have broken up the page HTML documentation into sections. All changes are required and mandatory unless specifically noted as optional. In other words... if you don't make the changes, your page won't work correctly.

NOTE: All code examples are shown in English or examples that would be used on a US/EN page. Remember you must change the locale information to your proper language and country codes and all labels and text to the proper translations for your page locale.

Doctype and root HTML element

The doctype for HTML5 is much more simplified. Since it's no longer XHTML (HTML with XML-type rules applied), there is no DTD nor any type that needs to be declared except that it's "html". As well, there is no more need for a namespace declaration nor any XML declaration. The root <html> element simply states what language the page is in.

v17 was like this (using your appropriate locale "$lc-$CC" value):

<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Strict//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-strict.dtd">
<html xmlns="http://www.w3.org/1999/xhtml" lang="en-US" xml:lang="en-US">

v18 is this:

<!DOCTYPE html>
<html lang="en-US">

<head> - Meta tags

In HTML5 there are new rules for meta tags. Most of the meta tags have been removed and the page's meta data has moved into the new "digitalData" JavaScript object variable. This new JS object and standard method to set meta data about your page is infinitely extensible. In addition and to support this, HTML5 no longer allows you to invent your own meta tags. There is a limited set of allowed meta tags.

Here are the only meta tags that are required to be put on a v18 page. All are required.

<head> - New meta data object

As mentioned, there are only a few meta tags that are used and most/all of the page's meta data is put into a new web standard JavaScript object variable called "digitalData". The digitalData object has a collection of specific standardized preset reserved properties (meta data), but also allows for an unlimited amount of custom meta data properties. Here is a summary of what the digitalData object is:

A standard data object that represents common data elements in a standardized way will allow development teams to implement design structures that populate the standard data object. Vendor code placed on the page would reference that standard structure, simplifying the process of onboarding or changing vendors, and shaving off expensive development cycles.

The proposed standard data object is a JavaScript object because of ubiquitous support for JavaScript in web browsers and web-based applications, as well as in other forms of digital properties like mobile devices, kiosks, and other digital content. The use of a JavaScript object means that the data is not embedded in the HTML markup and would not affect rendering of a page or performance. Developers will need only to populate the data fields in the object that are applicable to the page, keeping the size and complexity of the object to a minimum.

The standardized/reserved properties of the digitalData object are the "page.category" property and "page.pageInfo" children. These are the reserved/preset industry standard meta data properties. The "pageInfo.ibm" property is ours (IBM) where any and all IBM-specific and custom "made up" meta data can go.

<head> - v18 JS & CSS files

There are three required v18 core files needed on every www v18 page. They must be included in this specific order:

  • The new web metrics script
  • The core v18 CSS
  • The core v18 JS
<script src="//1.www.s81c.com/common/stats/ida_stats.js"></script>
<link href="//1.www.s81c.com/common/v18/css/www.css" rel="stylesheet">
<script src="//1.www.s81c.com/common/v18/js/www.js"></script>

NOTE: For v18 w3 pages, the stats file URL will be different.

In order to maximize the speed and efficiency of v18, lesser used page elements' CSS & JS have been omitted from the core JS & CSS. These were not chosen solely on usage, but on a complex formula called "is the juice worth the squeeze." In other words: How much KB will the element's CSS & JS add to the overall file size + how often is it used + convenience of including it in the core file - extra bloat for pages that don't use it and mobile users.

We ended up with only a handful (currently) of secondary optional CSS and JS files that you need to include based on the elements that you are using on your page.

Code syntax highlighter widget:

This enables the widget that color-codes your <pre> tag contents (code) and allows you to number the lines, highlight specific lines, and allow the user to select all and copy the code block.

<link href="//1.www.s81c.com/common/v18/css/syntaxhighlighter.css" rel="stylesheet">
<script src="//1.www.s81c.com/common/v18/js/syntaxhighlighter.js"></script>

Data tables or table widgets:

These are needed if you are using any data tables on your page and using the alternate highlight or styles, or any of the table widgets available like row selector, table scroll, sorting, filtering, pagination, etc.

<link href="//1.www.s81c.com/common/v18/css/tables.css" rel="stylesheet">
<script src="//1.www.s81c.com/common/v18/js/tables.js"></script>

Dynamic tabs:

Required for either the primary tab navigation or content space dynamic tabs widget. Enables the tabs to act like a basic show/hide and show the tab contents of the selected one.

<script src="//1.www.s81c.com/common/v18/js/dyntabs.js"></script>

Forms or form widgets:

If you have any forms on your page, including the "pull-down navigator" select list, these are required for styling and widgets to work.

<link href="//1.www.s81c.com/common/v18/css/forms.css" rel="stylesheet">
<script src="//1.www.s81c.com/common/v18/js/forms.js"></script>

Mustache.js:

This is an implementation of the mustache template system in JavaScript. Front-end web apps with MVC type setups will benefit from this. Makes it easy to AJAX an HTML template and replace variables in the template with content from a data object. Example: http://mustache.github.io/#demo. In other words, make complex use of JavaScript's replace() function easy. If you are doing a lot of templating or complex replace()s where you need some logic, this will help you.

<script src="//1.www.s81c.com/common/v18/js/mustache.js"></script>

<head> - Other misc elements

The <title>, <script>, and <style> elements still apply and work the same, but you do not need the @type attribute on <script> and <style>.
For <link> elements that embed CSS files, do not use the @title attribute. It has a special meaning and associated functionality.


<body> - Page start

The v18 page shell HTML is very very similar to v17. The same basic classes and IDs still exist, however there are some new HTML5 wrappers, new accessibility attributes, and other minor changes that are needed to make a proper v18 page. A fundamental change for v18 is that proper accessibility tagging must be coded in the HTML, or in the case of JS widgets injected as part of the dynamic widget or HTML that is dynamically generated on the page. In other words... whom/whatever generates the HTML is responsible for providing proper accessibility tagging and features for it.

Another thing you will notice is the lack of hidden <span>s used for accessibility. This is an old hack that is no longer needed with proper accessibility tagging. The new HTML and accessibility updates for v18 are cleaner and make the page more efficient.

<body> - stats.js/ntpagetag.js

#ibm-metrics: IBM is migrating from Netinsight to Coremetrics for web analytics. The metrics div has been removed from the bottom of the HTML for v18 www pages and a new script is used at the top of the page in the <head> (see above: v18 JS & CSS files).
w3 pages will be required to have both scripts on the page until Netinsight is completely sunset sometime near July. At that time, w3 pages will follow suit and no longer use the ntpagetag.js at the bottom of the page.

  • For v18 www pages, stats.js is not used.
  • For v18 w3 pages, both ida_stats.js (see above: v18 JS & CSS files) and ntpagetag.js are required until sometime near July. At that time ntpagetag.js will no longer be used on w3 pages and Netinsight will officially be sunset.
